## Step 1: Deploy the Database Schema

1. Go to your Supabase dashboard: https://supabase.com/dashboard/project/kruwfhzfqieuiuhqlutt
2. Open the **SQL Editor**
3. Copy the contents of `supabase/migrations/0001_blue_horizon_schema.sql`
4. Paste and **Run** — this creates all tables (students, teachers, admins, scores, timetables, assignments, messages, live_sessions, subscriptions, attendance) with Row Level Security enabled

## Step 2: Deploy the Edge Functions

Edge functions handle all sensitive operations (auth, registrations) using the **service_role key** (which bypasses RLS). They NEVER expose the service_role key to the browser.

### Install the Supabase CLI (if not installed)
```bash
npm install -g supabase
```

### Link your project
```bash
cd supabase
supabase login
supabase link --project-ref kruwfhzfqieuiuhqlutt
```

### Set the service_role key as a secret (REQUIRED)
```bash
# Set the service role key (used by edge functions to bypass RLS)
supabase secrets set SUPABASE_SERVICE_ROLE_KEY=your_service_role_key_here

# Also set the Supabase URL
supabase secrets set SUPABASE_URL=https://kruwfhzfqieuiuhqlutt.supabase.co
```

### Deploy all edge functions
```bash
supabase functions deploy auth-login
supabase functions deploy admin-register-student
supabase functions deploy admin-register-teacher
supabase functions deploy get-student-dashboard
supabase functions deploy send-message
supabase functions deploy get-messages
```

## Step 3: Seed an Admin Account

After deploying the `auth-login` function, you need to create the first admin. Since passwords are hashed, use this approach:

1. Open the Supabase **SQL Editor**
2. Run this SQL (replace with your desired admin username):
```sql
-- First, get a password hash by calling the auth-login function
-- OR insert directly with a pre-computed hash.
-- The hashing uses PBKDF2 (SHA-256, 100000 iterations).
-- For a quick start, you can use the Supabase dashboard to insert
-- a row into the admins table with a hash generated by the edge function.

-- Example (after deploying functions):
-- Call the auth-login function with a temp script to generate a hash,
-- then insert:
INSERT INTO public.admins (username, full_name, password_hash)
VALUES ('admin', 'School Administrator', 'PASTE_HASH_HERE');
```

**Easier method:** Create a temporary "seed-admin" edge function that inserts the first admin, then delete it.

## Step 4: Enable Realtime

The messenger uses Supabase Realtime for live message delivery. In the Supabase dashboard:

1. Go to **Database → Replication**
2. Ensure `messages` and `live_sessions` tables are added to the `supabase_realtime` publication (the schema SQL does this, but verify)

## Security Notes

- ✅ The **anon key** is in frontend code — this is safe (RLS protects data)
- 🔒 The **service_role key** is ONLY in Supabase secrets (set via `supabase secrets set`) — NEVER in frontend code or git
- 🔒 The **database password** is NEVER in the repo — only in `.env` (gitignored) and Supabase dashboard
- ✅ All user registration goes through edge functions (admin-authenticated)
- ✅ No self-signup — only admins create accounts
- ✅ Passwords are hashed with PBKDF2 (SHA-256, 100k iterations)

## Login Flow

1. User goes to `login.html`
2. Selects role (Student / Teacher / Admin) + enters username/password
3. Frontend calls `auth-login` edge function
4. Edge function checks credentials against the appropriate table (students/teachers/admins)
5. If invalid → shows: *"Please ensure your credentials are correct or ensure you are registered by your school admin. If not registered, contact your school admin. You can also contact your school admin for a change in credentials."*
6. If valid → returns user data, frontend stores session in a cookie and redirects to the dashboard

## Why students/ is SHARED (not copied to each class)

The `students/` folder is 290 MB (CBT engine, physics simulations, subject content). Copying it to 6 class folders would create a 1.7 GB git repository — far exceeding GitHub's recommendations and making deployment extremely slow. Instead, `students/` lives at `blue-horizon/students/` (shared) and all class dashboards reference it via relative paths (`../../students/`). Each class folder contains only class-specific files (dashboard, messenger, teachers).
